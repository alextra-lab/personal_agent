# ADR-0023: Qwen3.5 Model Integration — Thinking Control and Response Parsing

**Status**: Accepted
**Date**: 2026-03-03
**Deciders**: System Architect
**Related**: ADR-0003 (Model Stack), ADR-0008 (Hybrid Tool-Calling Strategy), ADR-0009 (Streaming vs Non-Streaming)

---

## 1. Context

### Problem

Qwen3.5 models (`qwen3.5-4b-mlx`, `qwen3.5-9b`) are configured in `config/models.yaml` as the `standard` and `reasoning` roles but are not usable in the agent due to two blocking failures:

1. **Thinking text in response content**: The `<think>...</think>` reasoning block generated by Qwen3.5 before every response is returned verbatim in the `content` field. The user sees raw XML-like thinking text mixed with the actual answer.

2. **Tool calls broken**: The `parse_text_tool_calls` fallback parser receives `content` that contains `<think>...</think>` blocks followed by the actual tool call text. The parser either confuses think-content for a tool call, or fails to find the `[TOOL_REQUEST]` block reliably, resulting in zero tool calls detected. The agent cannot execute any tools with the Qwen3.5 models loaded.

### Current Behaviour

`adapt_chat_completions_response` in `llm_client/adapters.py` hardcodes `reasoning_trace = None` (line 229) and never strips `<think>...</think>` content from the `content` field. This was correct for Qwen3 (which used a `/no_think` prompt suffix to suppress thinking) but incorrect for Qwen3.5 (which emits thinking tokens regardless until explicitly disabled at the API or chat-template level).

### Thinking Control: Qwen3 vs Qwen3.5

| Aspect | Qwen3 (previous) | Qwen3.5 (new) |
|---|---|---|
| Default behaviour | Non-thinking (fast) | Thinking on by default |
| Soft disable | Append `/no_think` to prompt | `/no_think` suffix still works |
| Hard disable | N/A | `chat_template_kwargs: {"enable_thinking": false}` via `extra_body` |
| Token budget | N/A | `thinking_budget` via `extra_body` |

The existing global `/no_think` suffix injection (in `orchestrator/executor.py`) was designed for Qwen3. For Qwen3.5, the preferred mechanism is a hard API-level disable via `extra_body`, which prevents the model from emitting `<think>` tokens at all, making the prompt suffix approach redundant and the stripping unnecessary for the `standard` model.

### Recommended Sampling Parameters

Qwen3.5 documentation specifies different optimal parameters per mode:

| Mode | temperature | top_p | top_k | presence_penalty |
|---|---|---|---|---|
| Thinking (reasoning) | 0.6 | 0.95 | 20 | — |
| Non-thinking (standard) | 0.7 | 0.8 | 20 | 1.5 |

`top_k` and `presence_penalty` are not in the standard OpenAI chat completions spec. They must be passed via `extra_body` for vLLM/LM Studio backends.

## 2. Decision

### 2.1 Strip `<think>` Tags in the Response Adapter (Immediate Unblock)

Add `_strip_think_tags(content: str) -> tuple[str, str | None]` to `adapters.py`. This function:
- Extracts all `<think>...</think>` blocks and joins them into `reasoning_trace`
- Returns the remaining text (trimmed) as the cleaned `content`
- Handles malformed input: unclosed tags, multiple blocks, empty blocks

Call it from `adapt_chat_completions_response`, replacing `reasoning_trace = None`.

This is a backend-agnostic fix. It works regardless of LM Studio/vLLM backend and regardless of whether `disable_thinking` is set. It is the minimum change needed to make Qwen3.5 produce correct output.

### 2.2 Add Per-Model Thinking Control to `ModelDefinition`

Add five new optional fields to `ModelDefinition` in `llm_client/models.py`:

```python
disable_thinking: bool          # Hard disable via chat_template_kwargs (default: False)
thinking_budget_tokens: int | None  # Token budget cap for thinking (default: None = unlimited)
top_p: float | None             # Top-p nucleus sampling (default: None)
top_k: int | None               # Top-k sampling, passed via extra_body (default: None)
presence_penalty: float | None  # Presence penalty (default: None)
```

`disable_thinking` and `thinking_budget_tokens` are mutually exclusive: a model cannot both have thinking disabled and a thinking budget.

### 2.3 Inject `extra_body` and Sampling Params in Request Builder

Extend `build_chat_completions_request` in `adapters.py` to:
- Accept the five new model-level fields
- Inject `top_p` and `presence_penalty` into the top-level payload (standard OpenAI fields)
- Build an `extra_body` dict containing `top_k`, and either `chat_template_kwargs: {"enable_thinking": false}` or `thinking_budget: N`, as appropriate

### 2.4 Guard `/no_think` Suffix Injection in `executor.py`

The existing `/no_think` suffix injection in `step_llm_call` is only effective for Qwen3-style models. When `disable_thinking=True` is configured for a model, thinking is already controlled at the API level. Injecting the suffix is redundant and adds noise to the conversation history. Guard the injection:

```python
uses_api_thinking_control = model_def is not None and model_def.disable_thinking
if tools and not uses_api_thinking_control:
    request_messages = _append_no_think_to_last_user_message(...)
```

### 2.5 Update `models.yaml` with Qwen3.5 Config

Apply recommended parameters for both active Qwen3.5 roles:
- `standard`: `disable_thinking: true`, non-thinking sampling params
- `reasoning`: `thinking_budget_tokens: 2048`, thinking sampling params

## 3. Alternatives Considered

### Alternative 1: Prompt-level `/no_think` suffix only (no API changes)

Use the existing suffix injection for Qwen3.5 without any code changes. Add `/no_think` to every prompt.

**Rejected** — Qwen3.5 thinks by default and the suffix is a "soft" control; the model may still emit `<think>` tokens on some prompts. Does not solve the content-pollution problem for the `reasoning` model (where thinking is desired). No way to add `top_k`, `thinking_budget`, or `presence_penalty`.

### Alternative 2: Tag stripping only (no per-model config)

Strip `<think>` tags globally from all responses as the sole fix. Do not add per-model thinking control.

**Rejected** — Stripping resolves the immediate symptom but does not prevent the model from burning thinking tokens on every request, including simple routing and tool-calling. The `standard` model (4B, used for fast tool orchestration) should never be spending tokens on reasoning. Token waste degrades latency and defeats the purpose of the role split.

### Alternative 3: Custom chat template at the server level

Start LM Studio / vLLM with a custom Jinja2 chat template (`qwen3_nonthinking.jinja`) that permanently disables thinking for the model, without any code changes.

**Rejected** — Requires server-side configuration change per deployment, is not self-documenting in the codebase, and cannot be toggled per-model at runtime when running multiple models on the same server. Does not generalise.

### Alternative 4: Separate model endpoints per thinking mode

Run thinking and non-thinking variants as separate model instances at different ports. Configure each role to point at the appropriate endpoint.

**Rejected** — Doubles memory pressure, doubles configuration complexity, and is operationally fragile. The `extra_body` API approach achieves the same result with a single model instance.

## 4. Consequences

### Positive

- Qwen3.5 models become fully usable in the agent (unblocks `standard` and `reasoning` roles)
- `<think>` content is correctly separated into `reasoning_trace` rather than polluting the user-visible response
- Tool calls parsed correctly after think-tag stripping clears the content field
- Per-model thinking budget prevents the `reasoning` model from over-generating thinking tokens on simple tasks
- Recommended Qwen3.5 sampling params improve output quality and reduce repetition (`presence_penalty`)
- Future models with similar thinking-on-by-default behaviour (e.g. future Qwen versions) can be configured without additional code changes

### Negative

- `build_chat_completions_request` signature grows by five parameters; mitigated by them being model-level (not call-level) and passed from config
- `extra_body` support for `chat_template_kwargs` is backend-dependent: confirmed for vLLM and SGLang; LM Studio MLX backend support requires empirical testing (see LM Studio Caveat below)
- `thinking_budget_tokens` for Qwen3.5 via LM Studio is unconfirmed; may be silently ignored by the backend

### Risk: LM Studio Backend Compatibility

The `chat_template_kwargs` mechanism is documented for vLLM and SGLang. LM Studio passes `extra_body` fields through to the underlying inference backend, but whether the MLX backend honours `chat_template_kwargs` is not guaranteed.

**Mitigation**: The think-tag stripping fix (Decision 2.1) is unconditional and backend-agnostic. Even if `disable_thinking` via `extra_body` is silently ignored, the response will still be cleaned correctly. The `disable_thinking` flag also serves as the guard for skipping the `/no_think` suffix — if `extra_body` is ignored, the suffix injection is still available by setting `disable_thinking: false` in `models.yaml`.

## 5. Implementation

### Files

| File | Change |
|---|---|
| `src/personal_agent/llm_client/adapters.py` | Add `_strip_think_tags`, call from `adapt_chat_completions_response`; extend `build_chat_completions_request` with extra_body |
| `src/personal_agent/llm_client/models.py` | 5 new fields on `ModelDefinition` with Pydantic validator |
| `src/personal_agent/llm_client/client.py` | Pass new model config fields to `build_chat_completions_request` |
| `src/personal_agent/orchestrator/executor.py` | Guard `/no_think` suffix injection |
| `config/models.yaml` | Updated `standard` and `reasoning` sections |

### `_strip_think_tags` contract

```python
def _strip_think_tags(content: str) -> tuple[str, str | None]:
    """Extract <think>...</think> blocks from model output.

    Returns:
        (cleaned_content, reasoning_trace)
        cleaned_content: content with all think blocks removed and stripped
        reasoning_trace: joined think block text, or None if no blocks found
    """
```

### `ModelDefinition` validator

```python
@model_validator(mode="after")
def _thinking_fields_exclusive(self) -> "ModelDefinition":
    if self.disable_thinking and self.thinking_budget_tokens is not None:
        raise ValueError("disable_thinking and thinking_budget_tokens are mutually exclusive")
    return self
```

### `models.yaml` target values

```yaml
standard:                          # Non-thinking fast model
  disable_thinking: true
  temperature: 0.7
  top_p: 0.8
  top_k: 20
  presence_penalty: 1.5

reasoning:                         # Thinking model with budget
  thinking_budget_tokens: 2048
  temperature: 0.6
  top_p: 0.95
  top_k: 20
```

## 6. Acceptance Criteria

- [ ] Agent response for a plain question shows no `<think>` text; `reasoning_trace` field is populated when thinking was present
- [ ] Agent successfully executes a tool call when using the `standard` (Qwen3.5-4b) model
- [ ] Agent successfully executes a tool call when using the `reasoning` (Qwen3.5-9b) model
- [ ] `disable_thinking=true` on the `standard` model results in `extra_body.chat_template_kwargs.enable_thinking=false` in the outgoing request payload (verified via debug log)
- [ ] `thinking_budget_tokens=2048` on the `reasoning` model results in `extra_body.thinking_budget=2048` in the outgoing request payload
- [ ] `top_k`, `top_p`, `presence_penalty` from `models.yaml` appear in the outgoing request payload for both roles
- [ ] `/no_think` suffix is NOT injected for the `standard` model (which has `disable_thinking=true`)
- [ ] `/no_think` suffix IS still injected for any model with `disable_thinking=false` (backward compatibility)
- [ ] All existing tests pass
