# ADR-0038: Context Compressor Model Selection

**Status:** Accepted
**Date:** 2026-03-30
**Deciders:** Project owner
**Depends on:** ADR-0033 (model taxonomy), ADR-0031 (model config consolidation)

## Context

The agent's context window management currently uses a static `[Earlier messages truncated]` marker when evicting older turns. This loses all information from evicted turns, degrading multi-turn conversation quality as conversations grow.

The Context Intelligence spec (Phase 4.1) calls for replacing this with compressed summaries generated by a lightweight LLM. This requires selecting a model for the `compressor` role — a new tier in the model taxonomy.

## Decision

Add a `COMPRESSOR` role to `ModelRole` and configure it with `gpt-5.4-nano` (OpenAI) as the default compressor model.

### Model Choice: Cloud Nano

| Criterion | gpt-5.4-nano (chosen) | Local sub_agent (qwen3.5-9b) | Local primary (qwen3.5-35b) |
|-----------|----------------------|------------------------------|----------------------------|
| Cost | $0.20/$1.25 per MTok | Free (local GPU) | Free (local GPU) |
| Latency | ~1-3s (network) | ~5-15s (GPU inference) | ~20-40s (GPU inference) |
| GPU contention | None | Shares GPU with primary | Blocks primary |
| Structured output | Strong | Good | Good |
| Concurrency | Up to 10 parallel | Limited by GPU VRAM | Single-threaded |

**Rationale:** The compressor runs asynchronously between turns. Cloud nano avoids GPU contention with the primary model (critical — compression must not delay user-facing inference). The cost is negligible: ~200 input tokens + ~100 output tokens per compression = ~$0.00017 per event.

### Summary Format: Structured Bullets

The compressor outputs a fixed-schema summary with four sections:
- **Decisions:** Choices made during the conversation
- **Entities:** Names, tools, technologies mentioned
- **Facts:** Established information
- **Open Items:** Unresolved questions

**Rationale:** Structured format enables downstream consumers (state document, recall controller) to parse specific sections. Prose summaries are harder to extract from.

### Token Budget: 200-400 Output Tokens

Max output tokens set to 512 (hard cap). The prompt instructs "maximum 200 words" (~260 tokens). This replaces potentially thousands of tokens of evicted messages with a ~300 token summary — a 3-10x compression ratio.

### Fallback: Graceful Degradation

If the compressor call fails (timeout, rate limit, error), the system falls back to the existing `[Earlier messages truncated]` marker. No user-visible degradation — the agent simply has less context, matching pre-Phase-4 behavior.

## Alternatives Considered

### 1. Local Sub-Agent Model
- **Pro:** No network dependency, no cost.
- **Con:** Competes for GPU with the primary model. Would need to wait for primary inference to complete, adding latency to the background compression task. The 9B model's summarization quality is lower than nano.
- **Rejected:** GPU contention is a hard constraint for this use case.

### 2. No LLM — Heuristic Extraction
- **Pro:** Zero cost, deterministic, fast.
- **Con:** Cannot understand conversational context. Would miss implicit decisions, nuanced entity relationships. Regex-based extraction is fragile.
- **Rejected:** Quality too low for the use case.

### 3. Primary Model with Lower Priority
- **Pro:** Best summarization quality.
- **Con:** Blocks all user-facing inference during compression. 35B model is single-threaded on current hardware.
- **Rejected:** Violates the async, non-blocking design constraint.

## Consequences

### Positive
- Evicted turns become summaries, preserving key context across long conversations.
- No GPU contention — compression is fire-and-forget on a cloud endpoint.
- Graceful fallback means zero regression risk.

### Negative
- Introduces a cloud dependency for a previously offline-capable feature. Mitigation: feature flag (`context_compression_enabled`) allows disabling.
- Small per-event cost. At ~$0.00017/event and <50 events/day, monthly cost is <$0.30.

## Acceptance Criteria

- [ ] `COMPRESSOR` role exists in `ModelRole` enum
- [ ] `compressor` model defined in `config/models.yaml`
- [ ] `compress_turns()` returns structured summaries from the compressor model
- [ ] `apply_context_window()` uses compressed summary when available
- [ ] Fallback to static marker on compressor failure
- [ ] Telemetry: `context_compression_completed`, `context_compression_failed` events
- [ ] Unit tests for compressor module and context window integration
