# Graph Identity Patterns for Multi-User AI Agent Memory

**Date**: 2026-05-09
**Author**: Research spike for FRE-213 (ADR-0052 amendment)
**Scope**: How do production graph-based agent memory systems model user identity in a shared knowledge graph?

---

## Question framing

Our system needs all three of the following simultaneously:

1. **Shared knowledge graph** — facts extracted from conversations are shared by default (e.g. "Paris is in France" or "Alice prefers Python") regardless of which user triggered the extraction.
2. **Per-user identity anchor** — each authenticated harness user (via Cloudflare Access) is a distinct, queryable entity in the graph, anchored by their stable `user_id` UUID — not by name.
3. **Per-user personal-context retrieval** — when a user asks "what did *we* talk about 5 days ago", the query should surface *their* conversations, not everyone's.
4. **Disambiguation invariant** — "harness user Alex" (`user_id` present) must never be confused with "friend Alex" extracted from conversation (no `user_id`).

---

## Systems surveyed

### 1. Letta (MemGPT)

**Architecture**: Hierarchical memory blocks injected into the LLM context window. Core blocks:
- `persona` — who the agent is
- `human` — what the agent knows about *the specific user it is talking to*

**Multi-user model**: Each user_id gets its own set of memory blocks. The Conversations API allows multiple agents to share a block (e.g. a shared world model) while keeping user-specific `human` blocks isolated. The block boundary is the isolation mechanism — there is no cross-user contamination because facts about "Alice" never appear in Bob's `human` block.

**Key insight for us**: Letta explicitly separates "facts about this user" (in the `human` block, scoped to user_id) from "general world facts" (in shared blocks). This is the shared-knowledge-plus-personal-context split we need. The implementation mechanism differs (blocks vs. graph nodes), but the design principle is identical.

**Identity vs. entity**: Not explicitly addressed in Letta's graph layer, but the block-level isolation prevents the collision problem.

---

### 2. Mem0 (Mem0g — graph variant)

**Architecture**: Directed labeled knowledge graph (Neo4j backend) plus vector embeddings per entity. Entities are nodes with type classification, embeddings, and a creation timestamp. Relationships are labeled directed edges.

**Multi-user scoping**: Four-scope hierarchy: `org_id > user_id > agent_id > run_id`. Memories tagged with `user_id` "belong to a specific user, persisting across all sessions." The `user_id` is **application-level auth** — not generated by the memory system. At retrieval time scopes compose; a query can retrieve all memories for a user across all runs, ranking `user_id` memories above session context above raw history.

**Disambiguation — explicitly documented**: The Mem0g paper states: *"Do not confuse character names mentioned in memories with the actual users who created those memories."* The system anchors identity through `speaker_id` (the auth-derived parameter), **not through conversation content**. User facts live in a per-user memory store; third-party entities live in the shared store.

**Anti-pattern they document**: Allowing the name-match deduplication path to merge a user's identity node with a same-named third-party entity mentioned in conversation. Mem0g avoids this by anchoring users on auth-derived IDs, not names.

**Key insight for us**: Mem0g is the closest analogue to our design. Our `user_id`-anchored `:Person` is equivalent to their `speaker_id`-anchored user node. The "do not confuse character names with users" guidance validates our disambiguation invariant.

---

### 3. Zep (Temporal Knowledge Graph)

**Architecture**: Bi-temporal KG. Three node types: Episodes (raw conversations), Entities (extracted concepts including speakers), Communities (clustered entity groups). Facts are edges with temporal validity windows.

**User modelling**: The speaker is extracted as an entity node in the same semantic subgraph as all other entities. **Zep makes no principled distinction between "system user" and "entity mentioned in conversation."** A user named "Alex" and a friend named "Alex" both become `:Entity {name: "Alex"}` nodes.

**Known limitation**: The lack of a `user_id` anchor means Zep cannot reliably answer "which Alex is the one who sent this message?" at query time. Fact ownership is not enforced.

**Relevant take-away**: Zep's flat entity model is simpler but breaks down for our use case. Their approach would create exactly the "harness user Alex vs. friend Alex" collision we are specifically trying to prevent.

---

### 4. ChatGPT memory (OpenAI)

**Architecture**: Single-user only. All memories stored as a flat list of "known facts" (key-value pairs). Injected verbatim into the system prompt on every turn. No graph — purely a flat list, fully materialised into the context window.

**Identity model**: One memory store per account (user_id = account). No concept of "shared knowledge" — all memories are personal to that account.

**Take-away**: Highly pragmatic for single-user (exactly what we are also starting with for the system prompt stanza). For multi-user, it doesn't apply — they just run separate accounts.

---

### 5. Neo4j multi-tenancy patterns

Three documented approaches:

| Approach | Mechanism | Shared entities | Fit |
|----------|-----------|----------------|-----|
| **Separate databases** (physical) | One Neo4j db per tenant | Not supported — no cross-db edges | ✗ Too heavy; breaks shared knowledge |
| **Label/property partitioning** (logical) | Tenant label or `user_id` property on all nodes; query-layer filter | Shared entities require either duplication or a third "common" database | △ Workable but operationally complex |
| **Provenance edges** | Tenant-neutral nodes; ownership modelled as edges from user to node | Naturally shared — edges record "who knows this" without copying the node | ✓ **Best fit** |

The provenance-edge approach (Pattern E below) is the community-recommended path when shared entities exist and per-tenant retrieval of "things my tenant touched" is required. Neo4j community explicitly identifies this pattern for scenarios like "multiple tenants share Person nodes but each tenant has their own interaction history with those Persons."

---

### 6. W3C PROV / provenance modelling

Standard model: `(:Activity)-[:wasAssociatedWith]->(:Agent)` and `(:Entity)-[:wasDerivedFrom]->(:Activity)`. Applied to our context:

```
(:Person {user_id})-[:PARTICIPATED_IN]->(:Turn)-[:DISCUSSES]->(:Entity)
```

This pattern answers "what did *this user* talk about in the last 5 days?" naturally via graph traversal from the user's :Person node, without requiring `Turn.user_id` as a denormalised property. The shared entities stay shared; provenance is modelled as edges, not as per-user copies.

Our `(:Turn)-[:DISCUSSES]->(:Entity)` schema already implements the right side of this. The missing piece for personal-context retrieval is a provenance edge from the user's `:Person {user_id}` to their `:Turn` nodes.

## Pattern comparison

| Pattern | Strengths | Weaknesses | Fit for Seshat |
|---------|-----------|------------|----------------|
| **A — Separate stores per user** (Letta/Mem0 default) | Clean isolation; no collision risk | Shared facts duplicated; N graphs to maintain | ✗ Breaks shared knowledge |
| **B — user_id scoping in shared graph** (Mem0g org_id/user_id) | Shared knowledge stays shared; personal retrieval derivable; clean auth integration | Queries need visibility filters; dedup risk if user_id not anchored | ✓ **Our chosen approach** |
| **C — Speaker-as-entity, no user_id anchor** (Zep) | Unified schema; conversation provenance is natural | No principled distinction between system user and mentioned entity; collision risk | △ Partial — we add the user_id anchor that Zep lacks |
| **D — Label/property partitioning** (Neo4j multi-tenant) | Single DB; query transparency | Noisy-neighbor risk; shared entities hard to model | ✗ Not recommended for our case |
| **E — Provenance edges** (W3C PROV / Neo4j community) | Shared entities stay shared; personal retrieval = graph traversal from user node; fully auditable | Schema complexity; requires explicit edge creation per turn | ✓ **Ideal for personal-context retrieval ticket** |

## Recommendation

**The proposed schema for FRE-213 is confirmed correct and is the community-recommended pattern for shared-knowledge + per-user-identity + personal-context retrieval.** No amendment required to the schema. Two points to add to the ADR:

### Confirmed design decisions

1. `:Person {user_id}` anchoring (not name-matching) — directly matches Mem0g's `speaker_id` anchor. The "do not confuse character names with users" guidance from Mem0g validates our disambiguation invariant.

2. Shared knowledge graph, not per-user graphs — matches Letta's shared-blocks + Mem0's org_id-level scope pattern. Our visibility model (`PUBLIC` default) is already correct.

3. `is_owner=true` single-owner edge — simpler than Letta's Conversations API multi-owner pattern, appropriate for the current single-owner deployment.

### Enhancement for the personal-context retrieval ticket (FRE-???)

Add a `PARTICIPATED_IN` provenance edge from the user's `:Person` to each of their `:Turn` nodes (created/updated when a Turn is written). This is the pattern Neo4j recommends for multi-tenant shared-entity graphs and aligns with W3C PROV. It enables:

```cypher
MATCH (u:Person {user_id: $uid})-[:PARTICIPATED_IN]->(t:Turn)-[:DISCUSSES]->(e:Entity)
WHERE t.timestamp > datetime() - duration({days: 5})
RETURN e
```

without needing `Turn.user_id` as a denormalised property (avoids schema duplication). **Recommended over adding `user_id` to Turn.** Defer to the personal-time-window retrieval ticket.

### Pre-existing dedup risk confirmed as HIGH

Mem0g explicitly documented this exact anti-pattern and built a defence against it (`speaker_id` anchor). Our dedup code (`memory/dedup.py:83-86`, exact lowercase name-match → MERGE) does not yet have this defence. The person-dedup-hardening follow-up ticket is correctly prioritised as HIGH.

---

## Open questions (none blocking FRE-213 implementation)

1. **PARTICIPATED_IN edge creation point**: should the provenance edge be written when a Turn is saved (memory/service.py:334), or lazily on first personal-context query? Writing at Turn-save time is more reliable. Decide in the personal-context retrieval ticket.

2. **Community nodes (Zep pattern)**: Zep groups entities into Communities for topic clustering. Worth evaluating for our `second_brain/` layer if entity counts grow; not needed now.

3. **Mem0's conflict-detector pattern**: Mem0g marks obsolete relationships as `invalid` rather than deleting them (temporal reasoning). Our current approach deletes and re-writes. Could improve for the knowledge-quality follow-ups (FRE-265, graph quality stream).
