# Cost-gate rework: per-feature lanes → per-provider pool — deliberation note

**Date:** 2026-07-16
**Author:** cc-explore2 (ephemeral seat), at master's request, read-only
**Status:** Decision note for the adr seat — not an ADR. Supersedes/reworks part of ADR-0065 (Cost Check Gate).
**Inputs:** `docs/architecture_decisions/ADR-0065-cost-check-gate.md`, `src/personal_agent/cost_gate/{gate,policy,types}.py`, `config/governance/budget.yaml`, `config/model_roles.yaml`, `config/profiles/cloud.yaml`, `src/personal_agent/cost_gate/__init__.py` (`budget_role_for`), `src/personal_agent/llm_client/{litellm_client,cost_tracker}.py`.

---

## TL;DR

Don't do a flat per-provider pool. Do a **provider-scoped outer cap + a denial-semantics-based reserved floor**, not a role- or writer-based one. The owner's "pinned writers are all the same" gut is right about the *design principle* (no bespoke per-role exception list) but wrong about the *concrete claim* — today's system already has a live provider collision between `main_inference` and `captains_log`/`insights` (all three resolve to Anthropic Claude Sonnet), and a naive flat pool would silently remove the protection that currently prevents it. `entity_extraction` is not actually in the same risk class — it sits on OpenAI and is accidentally isolated from the user-facing turn today. The right cut is **`on_denial: raise` vs `on_denial: nack`** (a distinction `budget.yaml` already encodes), not "writer vs non-writer" or "pinned vs open."

---

## 1. Grounding facts (what's actually true today, not assumed)

1. **The gate already has a shared-pool mechanism.** The synthetic `_total` cap (`CapEntry.role == "_total"`) is checked on *every* reservation regardless of role (`BudgetConfig.caps_for`) — it's a global backstop today (`weekly _total: $50`). The rework isn't "introduce a shared pool," it's "change what dimension the shared pool is keyed on" (global → per-provider) and "decide what happens to the per-role floors that sit inside it."
2. **The per-role model is already uneven in practice.** Only `entity_extraction`, `captains_log`, `main_inference`, `skill_routing`, `study` have explicit cap rows in `budget.yaml`. `insights`, `promotion`, `freshness` have no role-specific cap at all — they ride only on `_total`. So "clean per-feature lanes vs shared pie" is a false binary; the current system is already a hybrid, and the design question is which cut of hybrid to land on, not whether to have one.
3. **`provider` is schema-plumbed but not enforcement-wired.** `reserve()`, `_lock_counters()`, and `budget_counters` already carry a `provider` column (reserved for v2, always `None` in v1) — but `CapEntry` (the cap-matching model) has **no `provider` field**, and `caps_for()` never filters by provider. If a caller started passing a real `provider` value today without further changes, it would create a **separate counter row per (role, provider)** while the *cap* checked against it is still the plain role cap — i.e., it would accidentally **widen** the effective budget per role (each provider gets its own bucket against the same cap_usd) rather than pool anything. This is the opposite of what "shared provider pool" needs, and it means deliverable #3 ("is the decoupling as simple as it looks") has a real answer: **no, not quite.**
4. **Monitoring is already decoupled and provider-aware, independent of the gate.** `api_costs` already carries `provider`, and `cost_tracker.py` already has `get_total_cost(provider=)` / `get_weekly_cost(provider=)`. Per-role telemetry (`MODEL_CALL_COMPLETED` role/model) is untouched by anything below. So **deliverable #3's monitoring half is confirmed simple** — the ledger and its read-side aggregation need zero changes. Only the *gate's* enforcement key is where the real work (and the one flagged gap) lives.
5. **`main_inference` is already provider-agnostic as a budget role.** `budget_role_for()` folds `primary`, `sub_agent`, `compressor`, `router`, `reasoning`, `standard` — and, by the "unknown name defaults to main_inference" rule, the not-yet-existing `artifact_builder` too — all into one `main_inference` cap, regardless of which provider ends up serving the call. This is a deliberate existing design choice (a new role flag can't silently bypass the gate), and it means the *role* layer already answers "how much has the user-facing path spent, wherever it went" — which is close to what "shares the provider pie, from the owner's perspective" actually wants. A pure per-provider replacement would throw this coherent view away and fragment it by provider instead.
6. **The pinned writers do NOT all sit on the same provider.** From `config/model_roles.yaml` + `config/models.cloud.yaml` + `config/profiles/cloud.yaml`:
   - `entity_extraction` → `gpt-5.4-mini` → **OpenAI**.
   - `captains_log`, `insights` → `claude_sonnet` → **Anthropic**.
   - `primary` (→ `main_inference`, cloud profile) → `claude_sonnet` → **Anthropic**. `sub_agent` → `claude_haiku` → **Anthropic**.
   - `embedding` → OVH-managed (separate provider entirely) and `reranker`/`reranker_fallback` → Voyage — **neither is in `budget.yaml`'s `roles:` list at all today.** They aren't cost-gated through this mechanism currently (managed/fail-open per MASTER_PLAN). Any "per-provider pool" design has to explicitly decide whether it now onboards these two providers or continues to exclude them — silently inheriting the current gap would be a real scope hole in the ADR.
   - Local/self-hosted (Qwen) calls are **not gated at all** (explicit ADR-0065 non-goal, no cost) — this is an existing pressure-release valve: a user worried about a background job burning the Anthropic pool can already switch their primary/sub_agent to a local candidate and be unaffected. Any rework should preserve this, not accidentally gate it.

   **So today, `main_inference` already shares a provider (Anthropic) with `captains_log` and `insights`.** `entity_extraction`'s isolation from the user-facing turn is currently accidental — a side effect of being pinned to a different provider (OpenAI) — not a property of "being a pinned writer." Collapsing to a naive per-provider pool would keep `entity_extraction` isolated (nothing else forces its way onto OpenAI, yet) but would **merge `main_inference` into the same pool as `captains_log`+`insights`**, which today have their own separate per-role ceilings specifically so a NACK/redeliver background loop can't touch the user-facing cap. That's the firewall the task brief describes, and it is a *live*, not hypothetical, collision.
7. **ADR-0119 makes this worse, not just theoretical.** Once `primary`/`sub_agent`/`artifact_builder` become freely user-selectable across vetted candidates (ADR-0119, currently building — FRE-879/880/888-892), `main_inference`'s provider is no longer a static config fact — it's a runtime selection. If the user picks an OpenAI candidate for `primary`, the collision moves: now `main_inference` shares a pool with `entity_extraction` on OpenAI instead of with `captains_log`/`insights` on Anthropic. **The provider a user-facing role lands on is about to become dynamic at the exact moment this ADR proposes keying enforcement on provider** — which is a strong argument for *not* hardcoding the reserved-floor logic to "Anthropic" or to any named provider, and instead keying it structurally (see §3).

## 2. Pressure-testing the owner's "pinned writers are all the same" framing

**Where the owner is right:** there's no principled reason to hand-maintain a bespoke exception list keyed on role *identity* (`if role in {entity_extraction, captains_log, insights, embedding, reranker, reranker_fallback}: special-case`). That list is already how ADR-0119's writer-pin guardrail is expressed for a *different* axis (model-selection eligibility), and duplicating it here as a second, cost-gate-specific taxonomy would be exactly the kind of parallel special-casing the config-management work is trying to retire.

**Where the concrete claim breaks:** "provider-gated like everything else" implicitly assumes each pinned writer's provider assignment is incidental and interchangeable. It isn't — `entity_extraction`'s OpenAI pinning is *functionally load-bearing today* (it's the reason it doesn't compete with the user's turn), while `captains_log`/`insights`'s Anthropic pinning is the reason they *do* compete and currently need their own caps to stay out of the way. Treating "pinned writer" as one bucket erases a distinction that's actually about **provider co-location + retry semantics**, not about being pinned. Two writers can be equally "pinned" and have opposite starvation risk depending on which provider they happen to be pinned to.

**The distinction that actually matters** is one the system already has a name for: `on_denial: raise` vs `on_denial: nack` (`config/governance/budget.yaml`). A `nack` role is safe to let run dry against a shared pool — denial just means "retry next window," invisible to the owner. A `raise` role denied is a user-visible failure (`503 BudgetDenied` for `main_inference`; the original incident ADR-0065 exists to prevent). **The reserved floor belongs to `raise` roles, not to "non-writer" roles or "open" roles** — and it should be enforced generically against *whichever provider that role's calls are currently resolving to*, not against a hardcoded "Anthropic" pool, because ADR-0119 is about to make that resolution dynamic.

## 3. Recommendation

**Flat shared-provider-pie vs priority-tiered:** priority-tiered, not flat. Concretely:

- Generalize the existing `_total` synthetic-cap mechanism from **global** to **per-provider** (`_total` per `(time_window, provider)` instead of one global `_total`). This becomes the owner-facing "one dial per provider" the framing wants, and it's a small, additive change to `caps_for()`/`CapEntry` (add a `provider` field; match `cap.provider IS NULL OR cap.provider == resolved_provider`), not a rewrite.
- **Keep per-role caps for `raise`-semantics roles** (today: `main_inference`, `skill_routing`, `study`) as a **reserved floor inside whatever provider pool they're currently resolving to**. This is the firewall. It does not need to be role-specific beyond "is this role's `on_denial` == `raise`" — a single structural rule, not a hand-maintained list.
- **`nack`-semantics roles** (`entity_extraction`, `captains_log`, `insights`, `promotion`, `freshness`) can safely fold into the shared provider pie with no individual ceiling, or a much looser one purely for observability/tuning purposes — a runaway loop there degrades to "retried later," which is the behavior ADR-0065 already designed for and telemetered (D6).
- This reframes deliverable #2's binary ("special treatment or fold into the pie") as **neither, cleanly**: don't special-case writers, and don't fold everything flat either — key the floor off `on_denial`, which every role already declares, so no new taxonomy is invented and the owner's "no bespoke exceptions" instinct is honored structurally rather than violated.

**Deliverable #3 (is the decoupling as simple as it looks):** Monitoring — yes, already true, zero changes needed (ledger + read-side already provider-aware and independent of the gate). Gating — **not** "just change the enforcement key." Two real pieces of work, worth naming explicitly in the ADR so they don't get discovered mid-implementation:
1. `CapEntry`/`caps_for()` need an actual provider-matching dimension; today's `provider` plumbing through `reserve()`/counters is a placeholder that would misbehave (silently widen budgets) if fed real values without this change.
2. The reserve-site ordering constraint: today, `role` alone is sufficient to reserve (the role's provider is a static config fact). Under ADR-0119's open-role selection, the gate needs the **already-resolved** model's provider, so `reserve()` must happen *after* model resolution for open roles, not before — a sequencing dependency between the two in-flight ADRs (this one and ADR-0119) that the ADR text should call out rather than assume away.

**Scope note for the ADR:** decide explicitly whether `embedding`/`reranker`/`reranker_fallback` (OVH, Voyage) are in scope for the provider-pool rework or stay excluded as they are today. Silence here just inherits today's gap rather than resolving it.

## 4. Answering the deliverables directly

1. **Flat vs priority-tiered:** priority-tiered — a per-provider outer pool + a reserved floor for `on_denial: raise` roles, keyed structurally off denial-semantics rather than role identity or provider name (so it survives ADR-0119 making provider choice dynamic).
2. **Pinned writers — special treatment or fold in:** neither as stated. Split by `on_denial`, not by "writer." `entity_extraction` needs no floor today (accidental OpenAI isolation) but would need the *same generic rule* as `main_inference` the day it and an open role collide on a provider. `captains_log`/`insights` need no *individual* floor of their own — they need `main_inference`'s floor to exist in whatever pool they share with it.
3. **Monitoring/gating decoupling:** monitoring — confirmed simple, no changes. Gating — flagged as not simple: `CapEntry` needs a real provider dimension (currently absent), and reserve-site provider resolution must happen after model selection for open roles, not before.
4. **Shape for the adr seat:** this note's §3 recommendation is ADR-ready — it names the exact schema change (`CapEntry.provider`), the exact structural rule (reserved floor keyed by `on_denial == raise`), the exact sequencing dependency on ADR-0119 (reserve after resolution), and the exact open scope question (OVH/Voyage in or out) that a full ADR needs to settle rather than assume.
