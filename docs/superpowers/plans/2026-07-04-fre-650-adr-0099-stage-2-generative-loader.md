# FRE-650 — ADR-0099 stage 2: generative loader (roles resolve from the matrix, role headers removed)

**Ticket:** [FRE-650](https://linear.app/frenchforest/issue/FRE-650) · **ADR:** [ADR-0099](../../architecture_decisions/ADR-0099-configuration-management-and-validation.md) §D1 (mechanism half) · **Blocked by:** FRE-649 (merged, #356) · **Blocks:** FRE-652 (stage 4, C4)

## Scope decisions (surfaced before coding — not silently assumed)

1. **`primary`/`sub_agent` stay OUT of this ticket's resolution mechanism.** The ticket's own consumer list (Scope item 4) names six consumers: the consolidator extractor, `CaptainLogManager`, insights engine, context compressor, memory embedding, reranker — i.e. `entity_extraction`, `captains_log`, `insights`, `compressor`, `embedding`, `reranker`. `primary`/`sub_agent` are conspicuously absent. Investigation confirms why: they are resolved by a **separate, pre-existing mechanism** — `ExecutionProfile` (ADR-0044, `config/profiles/{local,cloud}.yaml`, `resolve_model_key()` in `config/profile.py`), a **session-scoped runtime choice** (which brain a conversation uses), not a **deployment-config divergence** (which matrix/D1 governs). Touching that mechanism is materially riskier (it's the live chat hot path) and not requested. **Decision:** the matrix keeps `primary`/`sub_agent` as `divergence: allowed` intent-only entries (as stage 1 left them); the new generative resolver is never called for these two roles, and `get_llm_client`/`resolve_model_key` in `profile.py` are untouched.

2. **Compressor/reranker get matrix-backed resolution too, even though `divergence: allowed`.** Scope item 4 lists them as consumers regardless of forbidden/allowed. Today their "role" *is* their model key (`models["compressor"]`, `models["reranker"]` — no `_role:` header indirection), and the value legitimately differs by which profile's model-definition file is loaded (compressor: local `gpt-5.4-nano` vs cloud `gpt-5.4-mini`; reranker: local `Qwen3-Reranker-0.6B` vs cloud `Qwen3-Reranker-4B-mxfp8`, FRE-696). **Decision:** the matrix declares `local`/`cloud` values for these two roles (both equal to the role's own name, since the divergence is in the *definition* each file attaches to that key, not the key itself); the resolver still goes through the matrix so there is exactly one code path for all six consumers, and AC-2(b)'s "matrix absent → raise" guarantee covers them too.

3. **`entity_extraction`/`captains_log`/`insights`/`embedding` (forbidden) get a single `all:` value.** `all: gpt-5.4-mini` / `all: claude_sonnet` / `all: claude_sonnet` / `all: embedding`, matching the values FRE-649 already corrected into both `models.yaml` and `models.cloud.yaml`'s (now-being-removed) headers.

4. **The `divergent_forbidden_role` guard fixture's meaning changes — by design, not by accident.** Stage 1's fixture modeled "assignment drift" (the same role's header names a *different model key* per profile — `gpt-5.4-nano` local vs `gpt-5.4-mini` cloud). Once a `forbidden` role has exactly one `all:` value, that failure mode is **structurally impossible** — which is the entire point of D1 stage 2 ("assignment drift becomes unrepresentable"). What remains checkable for a `forbidden` role is **definition drift**: the *same* model key resolves to a *different* underlying `ModelDefinition` (id/provider/max_tokens/temperature) across profiles' `models:` mappings — the `gpt-5.4-nano`→`gpt-4o-mini` case the ADR names throughout. **Decision:** rewrite the fixture so `model_roles.yaml` declares `entity_extraction: { divergence: forbidden, all: gpt-5.4-mini }`, and the fixture's `models.yaml` and `models.cloud.yaml` both define a `gpt-5.4-mini:` key but with different `id:` values — this is the AC-3 proof for stage 2's actual semantics. Call this out plainly in the fixture's own comment header so a future reader doesn't mistake it for a regression.

5. **`config_guard.py`'s role-resolution source changes from "read the profile YAML's `<role>_role:` header" to "read the matrix's own `all`/`local`/`cloud` value."** Once headers are removed from the profile YAMLs (Scope item 2), `_resolve_role_header()` (which reads those headers, defaulting to `"primary"` when absent) silently returns `"primary"` for every role — the guard's forbidden-divergence and dangling-reference checks would go **quietly vacuous** (a false-green regression neither this ticket's ACs nor FRE-649's own tests would catch unless the FRE-649 tests are also updated — see decision 6). This file is not named in the ticket's scope list but is a structural consequence of it. **Decision:** replace `_resolve_role_header(profile_yaml, role)` with a `_resolved_role_model_key(role_cfg, profile)` that reads the *matrix entry itself* (`role_cfg["all"]` for forbidden, `role_cfg[profile]` for allowed) — no YAML-header reads at all. Delete `_resolve_role_header` (dead once nothing calls it — orphaned by this change, not pre-existing, so removing it is in-scope per the surgical-changes rule).

6. **Existing tests that assume the old mechanism must be updated, not left to bit-rot.** Two categories:
   - **Fixture rot:** `tests/personal_agent/config/fixtures/{divergent_forbidden_role,dangling_reference}/config/model_roles.yaml` currently declare `divergence` only (stage-1 schema); they need `all`/`local`/`cloud` values added to match the new schema (decision 4 for the first; the second just relocates its existing per-profile values — `local: gpt-9-ghost, cloud: gpt-5.4-mini` — from the removed YAML headers into the matrix, unchanged semantics).
   - **Consumer-level mocks:** `tests/personal_agent/test_process_role_indirection.py::test_process_role_keys_resolve_to_valid_models` calls `getattr(config, "entity_extraction_role")` — this attribute is being removed from `ModelConfig`; rewrite to call the new `resolve_role_model_key()`. `tests/test_second_brain/test_entity_extraction_contract.py` (4 sites), `tests/personal_agent/second_brain/test_session_summary.py` (2 sites), `tests/personal_agent/llm_client/test_pricing.py`, `tests/personal_agent/orchestrator/test_attachment_cost_gate.py` (2 sites), and `tests/personal_agent/memory/test_memory_access_events.py` (2 sites) all currently set `entity_extraction_role=`/`captains_log_role=`/`insights_role=` either as a `ModelConfig(...)` constructor kwarg or a `MagicMock().entity_extraction_role` attribute to steer which model a test exercises. Once consumers stop reading that attribute, these mocks become **silently inert** (pydantic ignores unknown kwargs; the mock attribute is just never read) rather than failing loud — the dangerous kind of test rot. **Decision:** grep-audit every one of these sites during TDD (Step 4 below) and repoint each to whatever seam the consumer now actually reads (monkeypatching `resolve_role_model_key` directly is the simplest, lowest-blast-radius seam — see Step 2).

7. **Where the new resolver lives and how it avoids a second implementation of profile-detection.** `config_guard.py` already has `resolve_active_profile(model_config_path, matrix, root)` (FRE-649, unit-tested) — reused as-is, not reimplemented. **Decision:** the new `resolve_role_model_key(role: str) -> str` lives in `config/model_loader.py` (the module already responsible for "load and validate model configuration," already the home of `load_model_config`/`ModelConfigError`) and imports `config_guard.{repo_root, load_matrix, resolve_active_profile}` — mirroring the existing lazy-import-to-avoid-cycles convention already used between these two modules (`config_guard` lazily imports `settings`; neither imports the other at module scope today, so a new module-level import from `model_loader` → `config_guard` introduces no cycle).

8. **Header removal is not limited to the 2 "active" files.** AC-2(a)'s grep (`grep -rEl '^(entity_extraction|captains_log|insights|compressor|embedding|reranker)_role:' config/*.yaml`) covers **every** `config/*.yaml`, not just the two `active_profiles` files. Six files currently carry the three headers: `models.yaml`, `models.cloud.yaml` (both in `active_profiles`), plus `models-baseline.yaml`, `models.benchmark-4b.yaml`, `models.benchmark-4b-f16.yaml`, `models.benchmark-8b.yaml` (eval/benchmark artifacts, stage-4's retirement targets, but not retired yet). **Decision:** strip the 3 header lines from all six files now — mechanical, no logic change, and required for AC-2(a) to hold before stage 4 exists. `models.medium.yaml` already carries no headers (per FRE-648's audit) — untouched.

## Codex plan-review (2026-07-04) — findings incorporated

**Verdict on removing the 3 `ModelConfig` fields:** confirmed correct call — keeping them would preserve a second assignable API surface that undercuts ADR-0099's "one hand-edited home" / no-silent-fallback requirement. But codex found the blast radius is **larger than this plan first scoped**:

- **Missed consumers — `scripts/eval/` harnesses (AC-7 relevant, not just AC-1):**
  - `scripts/eval/fre630_extraction_quality/harness.py:117` — `role = cfg.entity_extraction_role` → `resolve_role_model_key("entity_extraction")`.
  - `scripts/eval/fre435_memory_recall/harness.py:101-119,565` — `_PIPELINE_ROLES` + `_pipeline_role_values()` + `_check_model_config_fidelity()` compare `getattr(config, role)` across **two explicitly-loaded `ModelConfig` instances** (active eval config vs. `config/models.cloud.yaml`), not the globally-active `settings.model_config_path`. This is the FRE-645/AC-7 eval-fidelity check and cannot be rewritten as a bare `resolve_role_model_key(role)` call, because that function (as designed) reads the *global* active profile.
  - **Design change:** give `resolve_role_model_key` an explicit optional `config_path: Path | str | None = None` parameter (`None` → use `settings.model_config_path`, mirroring `load_model_config`'s own existing convention exactly). `_pipeline_role_values` becomes `{role: resolve_role_model_key(_matrix_role_name(role), config_path=config.config_path_used) for role in ...}` — or more simply, since the harness already has both paths as concrete values (`settings.model_config_path` and `_PROD_MODEL_CONFIG_PATH`), call `resolve_role_model_key(role, config_path=path)` directly for each of the two paths and compare. This also makes the function directly reusable for Step 6's AC-1 golden test (call with `config_path="config/models.yaml"` and again with `"config/models.cloud.yaml"`) instead of monkeypatching global settings.

- **AC-1 proof-map gap:** the ADR's AC-1 names all six consumer paths, including `compressor`/`reranker` — this plan's golden test (Step 6) only asserts cross-profile *equality* for the four `forbidden` roles, which is correct (equality is meaningless for `allowed` roles — they're *supposed* to diverge). **Addition:** Step 6 also gets a lighter "resolves without error, matches the active profile's own model entry" check for `compressor`/`reranker` under both profiles — proves the matrix-backed path works for `allowed` roles too, without asserting an equality that would be wrong.

- **AC-2(b) proof-map gap:** the plan's Step 2 test (`resolve_role_model_key` raises when the matrix is missing) is helper-level only. ADR AC-2(b) requires **the runtime consumers from AC-1** to raise, not just the helper. **Addition:** Step 6's golden test also includes one case with a matrix-missing fixture root, calling one real consumer (e.g. `entity_extraction.extract_entities_and_relationships` with `model_override=None`) and asserting it raises rather than silently falling back — not just `resolve_role_model_key` in isolation.

- **New guard check — matrix shape validation.** Codex flagged that a malformed matrix (a `forbidden` role carrying `local`/`cloud` instead of `all`, or an `allowed` role missing a `local`/`cloud` value) is exactly the kind of drift stage 1's `divergent_forbidden_role` fixture used to catch, and nothing in this plan catches it once that fixture is repurposed for definition-drift (decision 4). **Addition:** a new `check_matrix_shape` function in `config_guard.py` (policy-class finding) asserting, per role, that `divergence: forbidden` roles declare `all` (and *not* `local`/`cloud`), and `divergence: allowed` roles declare at least one of `local`/`cloud` (and *not* `all`) — wired into `run_all_checks`, with a new `tests/personal_agent/config/fixtures/malformed_matrix_shape/` fixture and test.

- **Cache correctness, confirmed sound with one addition:** caching the matrix by resolved root path (mirroring `_load_model_config_at_path`'s pattern) is correct; caching `resolve_role_model_key(role)` *by role alone* would be wrong since `settings.model_config_path` (or the new `config_path` param) varies across calls/tests. **Addition:** since tests will monkeypatch/point at fixture roots repeatedly within a session, the matrix-loader cache needs a `.cache_clear()`-capable seam (a module-level `functools.lru_cache`, same as `_load_model_config_at_path`) and the new tests must call it in a fixture teardown/`monkeypatch` context to avoid cross-test cache bleed — note this explicitly in Step 2's test file so it isn't a flaky-test landmine later.
