## Re-scan result (done during planning; corrected after codex review)

Dumped every gold entity's current `v2_type` (69 entities across 39 cases) and checked each name/context against the two new type definitions.

- No entity besides `Neuroplasticity Chapter` names an authored/read-to-understand work (the `TechnicalArtifact`-typed entities — FastAPI, Python, PostgreSQL, Neo4j, pandas, etc. — are all tools/software, correctly staying `TechnicalArtifact`).
- **Correction (codex plan-review caught this):** the initial pass also kept `Redshift` at `Phenomenon`, reasoning its gold context ("they measure redshift — the stretching of light...") described the *process*. Codex flagged that this context is near-verbatim the research note's own `Redshift` boundary-probe wording ("measure the redshift — how far the light has stretched...") which the note ruled `QuantityMeasure (boundary)` under its own disambiguation rule (*measured quantity/value → QuantityMeasure; the stretching process itself → Phenomenon*). Both texts foreground "measuring redshift," so the rule's own precedent applies here too — `Redshift` moves to `QuantityMeasure`, adjudicated (not a fresh 3-way split; a direct application of the owner-approved rule).
- `Diffraction Limit` stays `Phenomenon` — its context ("bounds how finely an optical system can resolve detail") frames the *constraint*, not a measured value, matching the rule's other branch. Unchanged.

So step 2's "re-scan" moves **three** entities total: `Neuroplasticity Chapter` → `KnowledgeArtifact`, `Wavelength` → `QuantityMeasure`, `Redshift` → `QuantityMeasure`.
