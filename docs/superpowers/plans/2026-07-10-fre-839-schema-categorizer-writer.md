**Ticket:** FRE-839 (Approved, stream:build1, Tier-2:Sonnet)

## Scope (from ADR D2/D3/D4, restated precisely)

1. **Schema** (forward-compatible from day one, in the isolated study Neo4j ‚Äî `docker-compose.study.yml`, FRE-838):
   - `Concept {id, canonical_name, embedding, kind, valence, arousal}` ‚Äî hub; `kind` = the **preserved** ADR-0109 10-type entity kind, copied as-is from the frozen corpus's `Entity.entity_type`, never re-decided here.
   - `Surface {name}` `-[:ALIAS_OF]->` `Concept` ‚Äî case-variant surfaces collapse to one hub.
   - Evidence layer (immutable, append-only): `Episode`, `Mention` (`(:Episode)-[:HAS_MENTION]->(:Mention)-[:REFERS_TO]->(:Concept)`), `MembershipAssertion` (`(:Mention)-[:PRODUCED]->(:MembershipAssertion)-{[:ABOUT]->Concept, [:PROPOSES]->Category}`, carrying `{proposed_confidence, model, prompt_version, seed, when}`).
   - Derived layer (recomputable): `(:Concept)-[:MEMBER_OF {membership_confidence, support_count, last_supported_at}]->(:Category)`, `(:Concept)-[:MENTIONED_IN {when}]->(:Episode)`.
   - **Schema-only in v0** (declared, never populated by this ticket): `(:Category)-[:SUBSUMES {strength}]->(:Category)` edge type; `RelationAssertion` (directional, replacing `RELATED_TO` ‚Äî its own arm/gold-set, FRE-840+); the alias/subsumes/uncertain decision-type slot on category-pair candidates (that's the *consolidator's* candidate pairs, FRE-842 ‚Äî this ticket only needs the future edge type to not require a migration when FRE-842 lands).
2. **Ingest categorizer:** an LLM reads the **full conversation** and, for the concepts already known to be discussed in it (from the frozen corpus's `Session-[:DISCUSSES]->Entity` edges ‚Äî FRE-838's export, not rediscovered here), emits `MembershipAssertion`s: 1‚Äì3 proposed categories per concept with confidence, in-context (encoding specificity, D3) ‚Äî never a decoupled post-hoc pass. Provenance (`model`, `prompt_version`, `seed`) is **Python-stamped**, never asked of the model (mirrors `entity_extraction.py`'s `_build_provenance` split).
3. **Accretion writer:** appends `Mention`+`MembershipAssertion`s (never overwrites); derived `MEMBER_OF` is recomputed from the enlarged assertion set after each conversation. A fresh conversation re-mentioning an existing concept resolves to the **same** hub via alias lookup and **deepens** (not replaces) its memberships.
4. **Prove AC-1 and mechanism-AC-2** by running the categorizer+writer over the real frozen corpus (102 sessions, `scripts/study/snapshots/snapshot_manifest.json` dated 2026-07-10, `skipped_relationships: 0` ‚Äî confirmed complete after FRE-838's dynamic-discovery fix) and reading the result via one Cypher aggregation.

**Out of scope (belongs to later tickets):** baseline harness/ADR-0104 reproduction (FRE-840), cue-set/gold data + full AC-2 hard-negative P/R (FRE-841), the offline consolidator/canonicalization/œÑ_merge sweep (FRE-842), v0 synthesis/AC-3/AC-4/AC-7/continuation gate (FRE-843).

## File-by-file plan

### 1. `scripts/study/schema.py` (new)
`apply_schema(driver: Neo4jDriver) -> None` (async) issuing idempotent (`IF NOT EXISTS`) statements:
```cypher
CREATE CONSTRAINT concept_id_unique IF NOT EXISTS FOR (c:Concept) REQUIRE c.id IS UNIQUE
CREATE CONSTRAINT surface_normalized_name_unique IF NOT EXISTS FOR (s:Surface) REQUIRE s.normalized_name IS UNIQUE
CREATE CONSTRAINT category_normalized_name_unique IF NOT EXISTS FOR (cat:Category) REQUIRE cat.normalized_name IS UNIQUE
CREATE CONSTRAINT episode_id_unique IF NOT EXISTS FOR (e:Episode) REQUIRE e.id IS UNIQUE
CREATE CONSTRAINT mention_id_unique IF NOT EXISTS FOR (m:Mention) REQUIRE m.id IS UNIQUE
CREATE CONSTRAINT assertion_id_unique IF NOT EXISTS FOR (a:MembershipAssertion) REQUIRE a.id IS UNIQUE
CREATE VECTOR INDEX concept_embedding IF NOT EXISTS FOR (c:Concept) ON c.embedding
  OPTIONS {indexConfig: {`vector.dimensions`: 1024, `vector.similarity_function`: 'cosine'}}
```
**Codex plan-review finding, incorporated:** uniqueness constraints go on a `normalized_name` property (lowercase+trimmed), not the raw `Surface.name`/`Category.name` ‚Äî the writer's lookups are case-folded, and a raw-string constraint can't back a case-folded `MATCH`/`MERGE` (would force an unindexed scan, FRE-838's exact N+1/scan mistake) and wouldn't stop `Health issue`/`health issue` duplicating as separate `Category` nodes either. Both `Surface` and `Category` carry `normalized_name` (the merge/lookup key) **and** `display_name` (first-seen original casing, cosmetic only).

Module docstring documents the **schema-only-in-v0** elements as comments, not constraints (nothing to constrain ‚Äî the types don't exist yet): `SUBSUMES` edge type (populated FRE-855/v1 arm F), `RelationAssertion` (separate arm, FRE-840+), the alias/subsumes/uncertain decision slot on future category-pair candidate edges (FRE-842). CLI entrypoint (`uv run python -m scripts.study.schema`) for manual application; also called at the top of `run_ingest.py`.

### 2. `scripts/study/writer.py` (new)
- `@dataclass(frozen=True) ProposedMembership { concept_name: str, kind: str, category_name: str, proposed_confidence: float }` ‚Äî the categorizer's per-concept output shape.
- `@dataclass(frozen=True) AssertionProvenance { model: str, prompt_version: str, seed: int, when: datetime }`.
- `_normalize(name: str) -> str` ‚Äî `name.strip().lower()`, the single shared normalization key used by both `Surface.normalized_name` and `Category.normalized_name` lookups/merges.

- **`resolve_concept_hub` ‚Äî revised after codex plan-review (the original kind-gating was self-contradictory: gating *all* merges on `kind` equality would fail to collapse the ADR's own named example, since prod tagged `Arterial calcification`/`Arterial Calcification` with *different* kinds ‚Äî that inconsistency is precisely the bug D2 exists to fix; but *never* gating on kind reopens the homonym risk AC-2 tests for). The two merge paths get **different** gating, matching *why* each one is safe:**
  1. **Exact case-insensitive match (`normalized_name` equality) ‚Äî kind-BLIND, always merges.** This is specifically the "same written token, different casing" case D2 names ‚Äî kind-gating it would defeat the point (case-variants are exactly where prod's kind got confused). When an exact-name merge lands on an *existing* `Concept` whose stored `kind` differs from the new surface's kind, the hub's `kind` is **first-write-wins** (mirrors `create_entity`'s existing `entity_type` first-write-wins convention, `memory/service.py:1327`) ‚Äî the incoming surface still gets its `Surface` node `-[:ALIAS_OF]->` the existing hub; its own kind is not silently discarded, it's recorded on the `Surface` node itself (`Surface.kind`) as the surface's own original preserved value, even though the shared `Concept.kind` reflects the first writer.
  2. **Embedding-similarity fallback (no exact `normalized_name` match) ‚Äî kind-GATED, same as `dedup.py`.** Vector search on `concept_embedding` **filtered to `Concept.kind = $kind`** (mirrors `dedup.py:153`) + the ALL_CAPS-identifier guard (`_is_allcaps_identifier`, duplicated as a 3-line pure function rather than importing a private helper across packages) ‚Äî above `settings.dedup_similarity_threshold` merges. This is where the real homonym risk lives (different strings, similar embeddings) and kind is the correct disambiguator here.
  3. Else, `CREATE` a new `Concept {id: uuid, canonical_name: surface_name, kind, embedding, valence: null, arousal: null}` + `Surface {normalized_name, display_name: surface_name, kind}-[:ALIAS_OF]->Concept`.
  4. **Known, documented limitation (not fixed by this ticket):** two *byte-identical* strings (same spelling, same case) referring to genuinely different things ‚Äî e.g. a hypothetical exact-case "Mercury"/"Mercury" for the planet vs. the software product ‚Äî are not distinguishable by case-fold at all; kind is the only signal, and prod's own entity creation is *already* name-keyed (`memory/service.py` `entity_id = effective_name`) so such an exact collision would already have collapsed to one prod `Entity` node before this ticket ever runs, independent of anything FRE-839 does. The full hard-negative pairwise precision/recall bar (including any *synthetic* identical-string hard negatives FRE-841's eval set constructs) is explicitly FRE-841/843's job, not this ticket's "mechanism half" claim ‚Äî stated plainly in the final ticket comment, not silently glossed over.

- `async def write_episode(session, *, episode_id: str, source_session_id: str) -> None`: `MERGE (:Episode {id: episode_id}) SET e.source_session_id = source_session_id`.
- **`write_mentions_and_assertions` ‚Äî revised after codex plan-review (batching).** Takes the **whole session's** resolved memberships at once ‚Äî `list[tuple[concept_id, kind, list[ProposedMembership]]]` ‚Äî and issues **one** `UNWIND`-batched Cypher statement for the entire episode (not a per-concept or per-membership loop): a single `UNWIND $rows AS row` builds every `Mention`/`MembershipAssertion`/`HAS_MENTION`/`REFERS_TO`/`PRODUCED`/`ABOUT`/`PROPOSES` edge in one round-trip, `MERGE`ing `Category` by `normalized_name` and `MERGE`ing `(:Concept)-[:MENTIONED_IN {when}]->(:Episode)` in the same statement. Returns the distinct `(concept_id, category_normalized_name)` pairs touched, for the batched recompute step below.
- **`recompute_member_of_batch` ‚Äî revised after codex plan-review (was a per-pair loop; now one batched statement per episode):**
  ```cypher
  UNWIND $pairs AS pair
  MATCH (c:Concept {id: pair.concept_id})<-[:ABOUT]-(a:MembershipAssertion)-[:PROPOSES]->(cat:Category {normalized_name: pair.category_normalized_name})
  MATCH (a)<-[:PRODUCED]-(:Mention)<-[:HAS_MENTION]-(ep:Episode)
  WITH c, cat, avg(a.proposed_confidence) AS membership_confidence,
       count(DISTINCT ep) AS support_count, max(a.when) AS last_supported_at
  MERGE (c)-[m:MEMBER_OF]->(cat)
  SET m.membership_confidence = membership_confidence,
      m.support_count = support_count,
      m.last_supported_at = last_supported_at
  ```
  Takes the full set of `(concept_id, category_normalized_name)` pairs touched by one episode and recomputes all of them in one round-trip. `membership_confidence` = simple mean of backing assertions (deliberately simple v0 aggregate ‚Äî a more sophisticated reinforcement curve is FRE-842's tuning concern; AC-1 only checks edge *degree* and *support_count*, not this number's precision). Never guesses at a delta from a stale read ‚Äî always a full recomputation from the current assertion set.

### 5. `scripts/study/ac_proof.py` (new) ‚Äî the AC-1 / mechanism-AC-2 Cypher aggregation + report

**Revised after codex plan-review ‚Äî the original draft's inference was unsound.** It argued degree ‚â•2 *implies* provenance-distinctness because "one episode's categorizer call only proposes each concept's categories once per episode." That premise is false against this plan's own categorizer design: the categorizer proposes **1‚Äì3 categories per concept in a single call** (¬ß3 above), so one episode alone can produce `MEMBER_OF` degree 2 or 3 for a concept with zero other conversations ever mentioning it. Degree and provenance-distinctness are therefore computed as **two independent conditions**, not one inferred from the other:

- **Eligible set E:** `Concept`s with `count(DISTINCT episode)` via `MENTIONED_IN` `>= 2` (concepts mentioned in ‚â•2 conversations ‚Äî this part of the ADR's operationalization is unaffected by the above).
- **Degree condition:** for each concept in E, `count(MEMBER_OF edges)` (distinct categories) `>= 2`.
- **Provenance-distinctness condition, computed independently, not derived from degree:** for each concept in E, `count(DISTINCT episode)` **across all its `MembershipAssertion`s, regardless of which category they back** `>= 2` ‚Äî i.e. the concept's total backing-conversation count, not the count restricted to one category's assertions. A concept whose 2 categories both trace back to the *same single* episode's one categorizer call fails this condition even though its degree is 2.
- **AC-1's ‚â•60% bar** is evaluated over the **conjunction**: fraction of E where degree `>= 2` **AND** distinct-backing-episode-count `>= 2`. Both conditions are computed by separate `WITH`-chained aggregations in the same query (not inferred), so a concept that clears degree solely via one chatty conversation is correctly excluded from the ‚â•60% count.
- Query shape (illustrative, exact Cypher finalized during implementation):
  ```cypher
  MATCH (c:Concept)-[:MENTIONED_IN]->(ep:Episode)
  WITH c, count(DISTINCT ep) AS episode_count
  WHERE episode_count >= 2
  WITH collect(c) AS eligible
  UNWIND eligible AS c
  OPTIONAL MATCH (c)-[m:MEMBER_OF]->(:Category)
  WITH c, count(m) AS degree
  OPTIONAL MATCH (c)<-[:ABOUT]-(a:MembershipAssertion)<-[:PRODUCED]-(:Mention)<-[:HAS_MENTION]-(ep2:Episode)
  WITH c, degree, count(DISTINCT ep2) AS backing_episode_count
  RETURN c.id, degree, backing_episode_count,
         (degree >= 2 AND backing_episode_count >= 2) AS meets_ac1_bar
  ```
  then aggregate `meets_ac1_bar` across all rows for the ‚â•60% figure and `degree` for the median.
- Runnable standalone (`uv run python -m scripts.study.ac_proof`) against the real populated sandbox; prints a JSON report (median degree, % meeting the conjunctive bar, E's size, a handful of example concepts at the median for spot-check legibility) ‚Äî this **is** the AC-1 proof artifact for the final ticket comment. The report is honest either way: a null result (the corpus/categorizer doesn't clear the bar) is reported as such, not reframed ‚Äî consistent with the ADR's own "a clean null is a valid, budgeted outcome."
- Mechanism-AC-2 spot-check: enumerate the known case-variant pairs the ADR itself names as the original symptom (`Arterial calcification`/`Arterial Calcification`, `halitosis`/`Halitosis` ‚Äî confirmed present as separate prod `Entity` nodes per the ADR's forensic read) and assert each pair resolves to the **same** `Concept.id` post-ingest (`MATCH (s1:Surface {normalized_name: $a})-[:ALIAS_OF]->(c1), (s2:Surface {normalized_name: $b})-[:ALIAS_OF]->(c2) RETURN c1.id = c2.id`) ‚Äî this is the "mechanism half of AC-2" the ticket claims, not the full hard-negative pairwise P/R bar (FRE-841/843).

## Risk / judgment calls flagged for review
- **Membership-confidence aggregation is a simple mean** ‚Äî deliberately unsophisticated for v0; AC-1 only needs degree/support-count, not a tuned confidence curve (FRE-842's territory).
- **Category names are case-folded (`normalized_name`) only, no fuzzy/semantic merge** ‚Äî that's FRE-842's Node-Similarity/KNN canonicalization job. **Codex plan-review flagged this as a real risk to AC-1 itself, not just a cosmetic scope-cut:** if the same real-world category surfaces under enough distinct free-text names across 102 conversations (no fuzzy merge to collapse them), support/degree could be artificially suppressed, and AC-1 could fail for a reason that's an artifact of this ticket's deliberate scope-cut rather than a genuine signal about whether categorical accretion helps. Mitigation adopted: (1) the `normalized_name` case-fold (above) removes the cheapest, highest-volume source of spurious scatter now; (2) `ac_proof.py`'s report includes both the case-folded numbers and, as a diagnostic side-note, the *raw* (pre-normalization) category-name cardinality, so a reviewer can see whether scatter is materially inflating the category count; (3) if the real run's AC-1 numbers come back weak, the final ticket comment states plainly whether category-name scatter looks like a plausible contributing cause (diagnostic, not a re-scored pass) ‚Äî full fuzzy canonicalization remains FRE-842's job, not retrofitted into this ticket to force a pass.
- **New `study` budget role/cap** ‚Äî needs an explicit owner nod before the costly real run (Step 7 above), separate from code-merge approval. Codex plan-review confirmed keeping this isolated from `entity_extraction`'s cap is the right call, not over-engineering.
- **`seed` is a run-identifier, not a proven determinism guarantee** ‚Äî no provider-level seed support exists in this codebase or (for Claude) in the API itself.
- **Alias-resolution kind-gating is asymmetric by design** (exact-match kind-blind + first-write-wins; embedding-fallback kind-gated) ‚Äî codex plan-review caught that uniform kind-gating would fail the ADR's own named case-variant example, and uniform kind-blindness would reopen homonym risk. The asymmetric rule resolves both, with one known, explicitly-deferred gap: byte-identical same-case homonyms (not part of this ticket's named scope) are left to FRE-841/843's fuller hard-negative test.
